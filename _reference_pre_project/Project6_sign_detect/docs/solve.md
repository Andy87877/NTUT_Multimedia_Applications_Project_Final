在 train_pytorch_yolov4tiny.py 中，Darknet 檔案表頭（Header）的長度少寫了 4 個位元組（一個 32 位元整數），這導致自走車在解讀檔案時發生了「位移」與「截斷」錯誤。

🔍 詳細技術原因：
表頭大小的定義差異：

標準 Darknet 權重檔的最前面是表頭數據（包含模型版本號與訓練影像數）。
在早期版本中，表頭只有 4 個 int32 整數（16 位元組）：major、minor、revision 和 seen（看過的影像張數）。
但在較新的 Darknet 版本（也就是 JetBot 上 yolo_to_onnx.py 所採用的標準）中，seen 這個變數被改用 64 位的長整數 (uint64_t) 儲存。一個 64 位長整數相當於兩個 32 位整數。
因此，自走車上的解析器預期檔案的最前面要有 5 個整數（共 20 位元組）的表頭。
導致的「位移與截斷」錯誤：

先前的 Python 匯出程式只寫入了 4 個整數（16 位元組）。
當自走車的 yolo_to_onnx.py 讀取該檔案時，強行讀取了 20 位元組當作表頭，這導致它把我們模型第一層的第一個權重數值（4 位元組）也當作表頭吞進去了！
如此一來，後面所有卷積層的權重讀取位置全部往後位移了一個數字。
當讀取到檔案最後一層時，因為前面的位移，檔案最後就剛好缺少了最後一個權重數值（4 位元組），因此解析器就噴出： TypeError: buffer is too small for requested array（意即：檔案內容太小，不夠填滿最後一層的陣列）。
🛠️ 我們的修正方法：
我已經將 train_pytorch_yolov4tiny.py 中的表頭寫入改為：

python

# 從 4 個整數改為 5 個整數 (多加一個 0 作為 seen 的 64 位元補位)

header = np.array([0, 2, 5, 320000, 0], dtype=np.int32)
這樣權重檔案的表頭就會是標準的 20 位元組，自走車讀取時的起點與終點便能 100% 精準對齊！目前本機正在快速重新生成修正後的權重，完成後即可上傳部署！
