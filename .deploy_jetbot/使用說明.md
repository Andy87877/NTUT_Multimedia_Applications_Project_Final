## 🛠️ 2. 模型轉換與編譯 (於 JetBot 終端機執行)

為了讓 JetBot 達到即時推論速度（> 10 FPS），必須在車端將 YOLO 模型編譯為 TensorRT 引擎。

1. **開啟終端機並連線至 JetBot** (SSH 登入)。
2. **切換到 `yolo` 編譯目錄**：
   ```bash
   cd ~/jetbot/notebooks/road_following_team_1/trt_yolv4-tiny-master/yolo/
   ```
3. **執行 ONNX 格式轉換**：
   ```bash
   python3 yolo_to_onnx.py -c 4 -m yolov4-tiny-416
   ```
   *註：`-c 4` 代表類別數量為 4 類。*
4. **執行 TensorRT 引擎編譯 (啟用 FP16 半精度加速)**：
   ```bash
   python3 onnx_to_tensorrt.py -c 4 -m yolov4-tiny-416
   ```
   *編譯約需 2-5 分鐘，完成後會在同目錄生成 `yolov4-tiny-416.trt` 檔案。*

## 🚗 3. 運行主程式 (`Final_team_1.ipynb`)

開啟瀏覽器登入 JetBot 的 **Jupyter Notebook** 網頁介面，進入工作目錄並點擊開啟 **`Final_team_1.ipynb`**。第一次部署時先執行 **Cell 2** 建立道路 TensorRT 模型，完成後重新啟動 Kernel，再依需求執行以下單元：

### 🏁 單元一：純道路跟隨測試 (調校 PID 參數)
* **目的**：只載入 ResNet-18 道路循跡模型與 224x224 相機，不受 YOLO 與 PyCUDA 干擾，便於單獨調校 PID 與轉向參數。
* **執行步驟**：
  1. 依序執行 **`A1` ~ `A5`** 單元格載入模型、設定滑桿並定義道路控制函式。
  2. 執行 **`A6`** 啟動自動駕駛，將車輛放上賽道進行測試與調參。
  3. **調參說明**：
     * **`Speed` (速度)**：預設 `0.15`。
     * **`P Gain` (比例項)**：預設 `0.10`，若修正太慢可調大；若車頭劇烈搖晃可微幅調小。
     * **`D Gain` (微分項)**：預設 `0.02`，若車頭產生左右 S 型擺盪，可小幅調整。
     * **`Motor Bias` (偏置)**：若左右輪有硬體偏差導致直行偏斜，可藉此滑桿進行微調。
  4. **切換單元前必做**：執行 **`A7. 安全停止單元一`** 單元格釋放道路模型、GPU cache、相機與馬達資源。

### 🚥 單元二：純交通路牌辨識測試
* **目的**：只載入 YOLOv4-tiny 偵測模型與 416x416 相機，測試號誌動作。自走車在此模式下只會以直線前進（不進行轉向修正），偵測到路標時執行對應的停走動作。
* **執行步驟**：
  1. 依序執行 **`B1` ~ `B5`** 單元格，完成 YOLO 載入、介面建立與路牌控制函式定義。
  2. 執行 **`B6`** 啟動號誌偵測，手持或放置號誌卡片在鏡頭前測試。
  3. **切換單元前必做**：執行 **`B7. 安全停止單元二`** 單元格釋放 YOLO 模型、GPU cache 與相機資源。

### 🏁🏁 單元三：合併最終版 (道路跟隨 + 路牌辨識)
* **目的**：雙模型並行推論。使用 416x416 相機流，推論 YOLO 路牌後，對影像進行即時下採樣 (224x224) 供 ResNet 進行轉向預測，為期末考評分的最終整合版本。
* **執行步驟**：
  1. 依序執行 **`C1` ~ `C5`** 單元格，完成雙模型載入、介面建立與整合控制函式定義。
  2. 執行 **`C6`** 啟動完整自動駕駛，自走車將會同時進行道路跟隨與路標反應。
  3. **切換/結束前必做**：執行 **`C7. 安全停止單元三`** 停止自走車並釋放雙模型、GPU cache 與相機資源。

## 🚥 4. 號誌控制動作說明 (採用 Roboflow 字母排序類別順序)

本專案完美契合期末簡報規定的行為指標：

| Class ID | 號誌名稱 | 影像畫面標記 | 觸發寬度門檻 | JetBot 對應動作 |
| :---: | :---: | :---: | :---: | :--- |
| **Class 0** | `blocked` | `ID:0` (紅色標籤) | $\ge$ `Alert Width` | **永久停止行駛**並鎖定；即使正在執行 Stop/Rail 定時停車，仍持續偵測並可立即升級為永久停止。 |
| **Class 1** | `pedestrian` | `ID:1` (青色標籤) | $\ge$ `Alert Width` | **減速行駛**（基礎速度自動下調為 0.7 倍），通過後自動恢復。 |
| **Class 2** | `rail` | `ID:2` | $\ge$ `Alert Width - 20`，最低 20 px | **原地停止 5 秒** 後繼續行駛；冷卻時間為 15 秒。 |
| **Class 3** | `stop` | `ID:3` | $\ge$ `Alert Width` | **原地停止 2 秒** 後繼續行駛；冷卻時間為 12 秒。 |

### 🛡️ 防誤判與防假路標機制 (拿滿 10% 分數關鍵)：
* **寬度過濾**：小於 `Alert Width` 的路牌（代表距離很遠，或是賽道旁的假路標貼紙）會被程式自動忽略，避免提早停車。
* **高信心度過濾**：推論信心度設為 `0.6`，只對高可信度的路牌卡片做出反應。
* **重複觸發保護 (Cooldown)**：Stop 觸發後使用 12 秒冷卻，Rail 觸發後使用 15 秒冷卻，避免車子尚未駛離路牌便再次停車。

## 🚨 5. 安全緊急停機與資源釋放守則

由於 Jetson Nano 對 GStreamer 相機資源有嚴格的單例限制，若未釋放即重新開啟會造成 **Resource busy** 或相機死鎖。
* ⚠️ **黃金守則**：每當您要從單元一、二或三切換至其他部分時，**必須執行該單元最下方的安全停止單元格**（例如 `A7`、`B7` 或 `C7`）。
* 如果安全停止 Cell 顯示相機停止或 CUDA 清理失敗，請重新啟動 Kernel；不要直接載入下一個單元，避免 GPU OOM 或建立第二個相機 pipeline。
* ⚠️ **Kernel 故障限制**：Notebook 內的安全鎖只能處理程式仍在執行時的例外。若 Kernel、作業系統或供電直接失效，callback 無法繼續呼叫 `robot.stop()`；首次實車測試應架高車輪，並準備實體電源開關或獨立硬體 watchdog。
* 如果相機已經卡死（預覽畫面全黑或出現報錯），請在 JetBot 終端機執行以下命令重啟相機守護行程：
  ```bash
  sudo systemctl restart nvargus-daemon
  ```
